# YEMA — Direction produit, expérience et système de design

> **Version de référence — doctrine visuelle et expérience produit**
>
> YEMA est une maison où vivent les langues.
> Deux univers, une seule maison : **Monde** et **Racines**.

# 1. Ambition

YEMA ne doit pas ressembler à une plateforme de cours générique, à un tableau de bord SaaS ou à une application copiée sur Duolingo.

YEMA doit donner l’impression d’un produit :

* éditorial ;
* culturel ;
* humain ;
* structuré ;
* chaleureux ;
* sérieux pédagogiquement ;
* suffisamment premium pour inspirer confiance à une famille, un étudiant, un professeur ou un centre de langues.

Le niveau de finition attendu correspond à celui d’un produit conçu sur mesure par une équipe composée de :

* direction artistique ;
* product design ;
* UX research ;
* content design ;
* frontend senior ;
* motion design ;
* accessibilité ;
* photographie et illustration ;
* QA fonctionnelle et visuelle.

La valeur premium de YEMA doit venir de la précision, pas de l’accumulation d’effets.

# 2. Doctrine produit

## 2.1 YEMA est une maison

Le langage de la maison structure toute l’expérience :

* entrer ;
* franchir le seuil ;
* choisir une voie ;
* trouver sa place ;
* apprendre ;
* écouter ;
* prendre la parole ;
* rejoindre une classe ;
* retrouver son foyer ;
* transmettre ;
* recevoir une attestation ;
* continuer son voyage.

Cette métaphore ne doit pas devenir décorative ou folklorique. Elle sert à rendre le produit compréhensible et humain.

## 2.2 Principes non négociables

1. **Aucune IA**, dans aucun package et aucun parcours.
2. Les audios sont enregistrés, validés et servis depuis un stockage statique.
3. L’écoute est accessible sans accompagnement humain.
4. L’oral corrigé, l’écrit libre et le feedback personnalisé sont assurés par un humain.
5. Les exercices fermés sont déterministes et autocorrigés localement.
6. Les examens blancs sont préparés et versionnés.
7. Les contenus libres sont corrigés par un professeur ou un coach lorsque le suivi est actif.
8. Les promesses commerciales doivent correspondre à des fonctionnalités réelles.
9. Aucune fausse donnée, faux professeur, faux message ou fausse statistique ne doit être présenté comme réel.
10. Les univers Monde et Racines partagent la même qualité, mais pas exactement la même expression.

# 4. Principes d’expérience

## 4.1 Une action principale par écran

Chaque écran doit répondre clairement à quatre questions :

1. Où suis-je ?
2. Qu’est-ce que je peux faire ici ?
3. Quelle est l’action principale ?
4. Que se passe-t-il ensuite ?

Les actions secondaires ne doivent pas rivaliser visuellement avec l’action principale.

## 4.2 Montrer avant de demander de payer

Le parcours découverte doit permettre à l’utilisateur de :

* entendre une vraie voix ;
* apprendre de vrais mots ;
* réussir un exercice ;
* comprendre la méthode ;
* ressentir une progression.

L’activation intervient après une expérience réelle, pas après une succession de promesses marketing.

## 4.3 Le premium vient de la retenue

Éviter :

* les gradients bleu-violet ;
* le glassmorphism généralisé ;
* les néons ;
* les animations permanentes ;
* les cartes imbriquées sans nécessité ;
* les badges partout ;
* les illustrations génériques ;
* les ombres lourdes ;
* les dashboards remplis artificiellement.

Privilégier :

* une typographie expressive ;
* des espacements généreux ;
* des bordures fines ;
* des matières discrètes ;
* des contrastes maîtrisés ;
* des transitions courtes ;
* une iconographie cohérente ;
* des données utiles ;
* de vrais contenus.

## 4.4 L’humain doit être visible

Lorsqu’un utilisateur possède un suivi, le professeur ou le coach ne doit pas apparaître comme un simple add-on administratif.

La relation humaine doit être matérialisée par :

* un visage réel ;
* un nom ;
* une courte présentation ;
* un prochain rendez-vous ;
* un devoir ;
* un feedback ;
* un fil de conversation ;
* un délai moyen de réponse ;
* une progression partagée.

# 5. Système typographique

## 5.1 Polices

### Fraunces

Utilisée pour :

* titres éditoriaux ;
* greetings ;
* mots de langue mis en valeur ;
* signatures d’univers ;
* messages de progression ;
* grandes étapes émotionnelles.

L’italique exprime l’âme et la chaleur.

Le romain exprime la culture, le contenu et la pédagogie.

### Manrope

Utilisée pour :

* corps de texte ;
* navigation ;
* boutons ;
* formulaires ;
* tableaux ;
* données ;
* métadonnées ;
* interfaces fonctionnelles.

### JetBrains Mono

Utilisation exceptionnelle :

* codes de classe ;
* numéros de référence ;
* éléments techniques ;
* données courtes nécessitant une lecture caractère par caractère.

Elle ne doit pas devenir une police décorative.

## 5.2 Échelle typographique indicative

| Usage              |   Mobile |  Desktop |
| ------------------ | -------: | -------: |
| Display éditorial  | 32–40 px | 52–72 px |
| Greeting dashboard | 27–32 px | 36–46 px |
| Titre écran        | 25–29 px | 34–42 px |
| Titre section      | 20–23 px | 24–30 px |
| Titre carte        | 16–18 px | 17–20 px |

Aucun texte essentiel ne doit être inférieur à 12 px.

Les textes à 11 px sont réservés aux métadonnées secondaires non critiques.

## 5.3 Rythme typographique

* Longueur optimale d’un paragraphe : 45 à 75 caractères.

* Les titres ne doivent pas être entièrement en capitales, sauf le wordmark et quelques labels courts.
* Les majuscules espacées sont réservées aux labels de territoire ou aux métadonnées éditoriales.

* disabled ;
* loading ;
* success ;

* loading ;
* empty ;

Les états ne doivent pas se limiter à une icône et une phrase générique. Ils doivent expliquer :

# 10. Mouvement

## 10.1 Principes

Le mouvement sert à :

* confirmer une action ;
* expliquer une transition ;
* montrer une progression ;
* orienter l’attention ;
* matérialiser une réussite.

Il ne sert pas à décorer constamment l’interface.

## 10.2 Durées

| Interaction             |              Durée |
| ----------------------- | -----------------: |
| Hover/focus             |         120–180 ms |
| Ouverture carte ou menu |         180–240 ms |
| Changement d’étape      |         240–360 ms |
| Célébration de leçon    | 500–900 ms maximum |

## 10.3 Contraintes

* aucune animation bounce générique ;
* pas de parallax important ;
* pas de texte animé en permanence ;
* respecter `prefers-reduced-motion` ;
* ne jamais retarder une action pour montrer une animation ;
* ne pas modifier les animations canoniques de la landing sans autorisation.

* éviter les formulaires longs ;
* ne pas demander une information qui ne sera pas utilisée ;
* permettre d’explorer l’autre univers ultérieurement.

# 12. Choix du territoire — étape 1/3

## Intention

Donner le sentiment de franchir deux portes différentes dans une même maison.

## Structure

* Fond seuil.
* Wordmark en haut.
* Indicateur discret « Étape 1 sur 3 ».
* Titre Fraunces :
  **« Par où commence ton voyage ? »**
* Texte :
  « Choisis l’univers qui te ressemble. Tu pourras explorer l’autre plus tard. »

## Portes

### Monde

* Accent brass.
* Signature : *le voyage*.
* Illustration ou pictogramme de mouvement, jamais un clipart d’avion générique.
* Promesse :
  « Étudier, travailler, partir. »
* Détail :
  « Allemand, anglais, français… vers une certification. »

### Racines

* Accent terracotta.
* Signature : *les racines*.
* Illustration organique ou symbole de transmission.
* Promesse :
  « Transmettre, retrouver, relier la famille. »
* Détail :
  « Swahili, lingala, douala… tes origines. »

## Interaction premium

Au focus ou hover :

* la surface s’éclaircit légèrement ;
* le symbole avance de quelques pixels ;
* la bordure prend l’accent du territoire ;
* une phrase secondaire peut apparaître sans déplacement brutal du layout.

Lien de secours :

« Tu hésites ? **Commence par le voyage.** »

# 13. Choix de la langue — étape 2/3

## Principes

* Aucun drapeau national.
* Utiliser monogramme, nom, région et système de certification.
* Une langue active ne doit pas être visuellement confondue avec une langue « bientôt ».
* Une langue indisponible doit être non interactive ou clairement marquée.
* Ne pas annoncer une date de disponibilité sans engagement réel.

## Monde

### Active

**Allemand**

* badge : « A1 découverte gratuite » ;
* sous-texte : « Goethe · A1 à C1 » ;
* courte phrase éditoriale :
  « Pour étudier, travailler et vivre dans l’espace germanophone. »

### Bientôt

* Anglais — TOEFL / IELTS ;
* Français — DELF / DALF.

## Racines

Avant publication, la liste des langues réellement actives doit être confirmée.

Chaque langue ouverte doit disposer de :

* contenu validé ;
* voix natives ;
* au moins quatre cours de découverte ;
* curriculum ;
* responsable éditorial ;
* variété linguistique identifiée lorsque nécessaire.

Ne pas afficher une langue comme ouverte uniquement parce que son nom existe dans l’interface.

* cartes radio entièrement cliquables ;
* bordure accent sur sélection ;
* coche animée subtile ;

## Monde

* A1 — Je pars de zéro ;
* A1 — J’ai quelques bases ;
* A2 — Je me débrouille ;

Le niveau sélectionné ne constitue pas une certification.

* Je découvre la langue ;

Le mapping vers É1–É5 doit rester invisible à cette étape, sauf lorsqu’il aide réellement l’utilisateur.

# 15. Leçon de découverte

## 15.1 Intention

L’utilisateur doit apprendre quelque chose dès la première minute.

La découverte ne doit pas être un faux aperçu marketing. Elle doit utiliser le même moteur pédagogique que les cours payants.

## 15.2 Structure générale

Quatre cours gratuits :

1. salutations ;
2. famille ou présentation ;
3. nombres ;
4. révision et fin de découverte.

Chaque cours contient :

1. contexte de la leçon ;
2. audio enregistré ;
3. transcript ;
4. traduction optionnelle ;
5. vocabulaire ;
6. mini-exercice ;
7. feedback immédiat ;
8. progression ;
9. prochaine étape.

## 15.3 Progression

Afficher une progression segmentée :

```text
1 / 4
2 / 4
3 / 4
4 / 4
```

La progression doit être :

* compréhensible sans couleur ;
* accessible aux lecteurs d’écran ;
* persistée ;
* reprise après fermeture ;
* non réinitialisée sans action explicite.

## 15.4 Lecteur audio premium

Le lecteur doit afficher :

* titre ;
* locuteur ;
* durée ;
* lecture/pause ;
* progression ;
* replay de 5 secondes si techniquement justifié ;
* vitesse 0,75× / 1× / 1,25× si utile ;
* transcript repliable ;
* badge « voix enregistrée ».

Ne pas simuler une waveform sans données audio réelles.

Ne jamais générer de voix dynamiquement.

## 15.5 Exercices

Mécaniques autorisées :

* choix multiple ;
* appariement ;
* texte à trou ;
* remise en ordre ;
* reconnaissance auditive ;
* saisie courte avec réponses déterministes ;
* classement.

Le feedback doit expliquer, pas seulement indiquer « correct » ou « incorrect ».

Les réponses alternatives acceptables doivent être préparées et versionnées.

# 16. Fin de découverte

## Structure

* progression 4/4 ;
* signe de réussite ;
* message dans la langue ;
* récapitulatif ;
* thèmes étudiés ;
* statistiques réelles ;

## Statistiques

Afficher uniquement des données réellement calculées :

* cours terminés ;
* mots rencontrés ;
* exercices réussis ;
* minutes d’écoute.

Éviter les statistiques artificiellement gonflées.

## Bloc de suite

Monde :

* aperçu du niveau A1 ;
* prochaines leçons ;
* examens blancs ;
* attestation.

Racines :

* aperçu de l’échelle É1–É5 ;
* récits ;
* veillées ;
* progression personnelle ou familiale.

# Suivi humain Racines et coaching carrière

## 1. Suivi coach de langue Racines

Le suivi coach Racines est un add-on humain disponible en complément d’un abonnement Solo ou Famille.

Il possède un **tarif unique**, indépendant :

* de la langue ;

## 3. Contenu du suivi

Le prix de 30 000 XAF / 45 € par mois comprend, pour une personne :

### Corrections humaines

* jusqu’à huit productions corrigées par mois ;
* maximum deux productions par semaine ;

* ou production écrite de 250 mots maximum ;
* feedback vocal ou écrit du coach ;
* explication des erreurs principales ;
* conseil concret pour progresser.

### Rendez-vous

* une session individuelle de 30 minutes par mois ;
* visioconférence ou appel audio ;
* travail sur la prononciation, l’expression et les difficultés constatées ;

* objectif mensuel ;
* activité conseillée ;
* suivi de la progression ;
* réponse aux productions sous 48 heures ouvrées lorsque la capacité opérationnelle le permet ;
* historique des corrections dans l’espace de l’utilisateur.

Aucune correction automatique ou intelligence artificielle ne doit être utilisée.

## 4. Positionnement

Le coach de langue Racines aide l’utilisateur à :

* prendre confiance à l’oral ;
* améliorer sa prononciation ;
* formuler des phrases ;
* comprendre le contexte culturel ;
* raconter ;
* participer à une conversation ;
* pratiquer dans son foyer ;
* transmettre la langue.

Le coach n’est pas un chatbot, un assistant automatique ou un support client.

C’est une personne réelle, sélectionnée et rémunérée pour son expertise linguistique et culturelle.

Aucun forfait « coaching illimité pour tout le foyer » ne doit être proposé pendant la bêta.

Une offre familiale de coaching pourra être étudiée après observation :

* du nombre moyen de productions ;
* du temps réel passé par les coachs ;
* de la demande des familles ;
* de la marge ;

# Amendement A — Stratégie mobile YEMA

## A.1 Mobile-first

YEMA est conçu **mobile-first**. La bêta doit proposer une application web responsive complète qui fonctionne prioritairement à **360 × 800 px** et **390 × 844 px**, en gardant desktop et tablette comme surfaces également valides mais secondaires dans la hiérarchie de conception.

Exigences bêta pour chaque parcours utilisateur :

* application web responsive complète ;
* fonctionnement prioritaire à 360 × 800 et 390 × 844 ;
* navigation tactile ;
* formulaires adaptés aux claviers mobiles ;
* audios fonctionnels sur Android et iOS ;
* dashboards sans scroll horizontal ;
* tableaux transformés en listes ou cartes lorsque nécessaire ;
* actions principales accessibles au pouce ;
* reprise de progression ;
* gestion correcte des réseaux lents ;
* états hors connexion explicites ;
* compatibilité future PWA.

La bêta ne nécessite pas encore une application native. Le web mobile doit d'abord prouver sa robustesse.

## A.2 PWA

La PWA doit être prévue **après stabilisation du cœur produit**, pas avant. Elle pourra inclure :

* installation sur l'écran d'accueil ;
* mode standalone ;
* cache de l'interface essentielle ;
* accès aux dernières leçons ;
* téléchargement explicite de certains audios ;
* écran hors connexion ;
* notification de mise à jour ;
* reprise de progression avant synchronisation.

**Ne pas mettre en cache sans protection appropriée** :

* profils enfants ;
* messages privés ;
* corrections humaines ;
* données de paiement ;
* sessions expirées ;
* contenus soumis à des restrictions de droits.

## A.3 Application native future — lot P7

Un lot futur est réservé :

```text
P7 — Application mobile YEMA
```

Il **ne commence qu'après** :

* stabilisation du web mobile ;
* validation de la rétention ;
* mesure de l'usage mobile ;
* stabilité de l'authentification ;
* stabilité des paiements ;
* stabilité des cours ;
* fonctionnement réel des coachs et professeurs ;
* validation des règles de protection des enfants.

Fonctionnalités prioritaires dans l'app native :

1. connexion et onboarding ;
2. choix Monde/Racines ;
3. cours et exercices ;
4. lecture audio ;
5. progression ;
6. mode hors ligne limité ;
7. notifications ;
8. envoi de productions ;
9. réception des corrections ;
10. espace foyer ;
11. profils enfants ;
12. rendez-vous professeur ou coach.

Les outils complexes centre, admin et création de contenu peuvent rester sur le web desktop.

## A.4 Critères mobiles obligatoires

Chaque parcours utilisateur doit :

* fonctionner à 360 et 390 px ;
* ne pas avoir de scroll horizontal ;
* avoir des cibles tactiles de 44 × 44 px minimum ;
* ne pas être masqué par le clavier ;
* gérer les safe areas iOS ;
* utiliser des unités viewport modernes ;
* préserver la progression ;
* gérer le retour arrière ;
* fonctionner sur réseau lent ;
* respecter `prefers-reduced-motion` ;
* ne pas imposer le desktop pour terminer une leçon.

Le breakpoint 360 px sert à vérifier la robustesse mobile (voir §7.1). Le web mobile n'est jamais un mode dégradé.

# Amendement B — Messagerie de suivi — professeur ou coach ↔ élèves

## B.1 Décision produit

La messagerie de suivi est :

* liée au suivi humain ;
* privée ;
* fermée ;
* contextualisée ;
* limitée à **une classe Monde** ou **un cercle Racines** ;
* 100 % humaine ;
* sans IA.

Pour la bêta :

```text
une classe Monde fermée
ou
un cercle Racines fermé
```

**Aucun espace communautaire transversal** entre plusieurs classes ou cercles.

## B.2 Fil unique

Chaque classe ou cercle possède un **fil chronologique unique**.

Le fil regroupe :

* annonces ;
* devoirs ;
* invitations orales ;
* notes vocales ;
* corrections ;
* messages texte ;
* réactions ;
* encouragements entre pairs.

**Ne pas créer pendant la bêta** :

* canaux multiples ;
* salons publics ;
* messages privés entre élèves ;
* messagerie générale ;
* espace transversal.

## B.3 Composition visuelle

### Professeur ou coach

* messages à gauche ;
* avatar réel ;
* nom ;
* rôle ;
* correction officielle identifiable.

### Utilisateur actuel

* messages à droite ;
* accent du territoire ;
* heure ;
* statut de remise ou de partage.

### Pairs

* messages à gauche ;
* nom visible ;
* avatar distinct ;
* couleur contrôlée et accessible.

**La position seule ne doit jamais identifier l'auteur pour l'accessibilité** — nom, avatar et rôle sont obligatoires.

## B.4 La voix comme cœur du suivi

Les notes vocales sont **l'interaction principale**.

Chaque note doit afficher :

* lecture/pause ;
* durée ;
* progression ;
* forme d'onde réelle ou dérivée du fichier ;
* auteur ;
* heure ;
* loading ;
* erreur ;
* réécoute.

**Ne pas afficher une fausse waveform décorative.**

Composer mobile — hiérarchie visuelle :

```text
bouton joindre
champ texte
grand bouton micro    ← visuellement prioritaire, accessible au pouce
bouton envoyer
```

Le bouton micro doit être **le plus grand** et le plus proche du pouce.

Limites bêta :

* note vocale élève : **3 minutes maximum** ;
* correction vocale : **5 minutes maximum** ;
* réécoute avant envoi ;
* annulation possible ;
* consentement explicite pour le microphone ;
* aucun traitement IA ;
* aucune analyse automatique ;
* aucun entraînement de modèle.

## B.5 Types de messages

Prévoir :

* annonce ;
* devoir ou invitation orale ;
* production vocale ;
* production écrite ;
* correction humaine vocale et textuelle ;
* message d'un pair ;
* réaction ;
* réponse contextualisée.

**La correction doit référencer la production concernée** (lien parent-enfant explicite).

## B.6 Variante Monde

Territoire :

```text
espresso
brass #C9A34E
```

Exemple d'en-tête :

```text
Klasse A1 · Deutsch
Herr Kwessi · 8 élèves
```

Registre :

* « à la classe » ;
* « Devoir oral » ;
* « remis » ;
* « correction » ;
* « prochaine étape ».

Exemple de message coach :

> Sehr gut ! Juste le *ch* de *ich* — plus doux.

## B.7 Variante Racines

Territoire :

```text
terre
terracotta #C77B54
```

Exemple d'en-tête :

```text
Ton cercle · Lingala
Coach Mokili · 5 proches
```

Registre :

* « au cercle » ;
* « Fais entendre ta voix » ;
* « partagé » ;
* « ton cercle » ;
* « écouter ensemble ».

Exemple de message coach :

> Malamu ! Ta grand-mère te comprendrait. Juste *ndeko* — appuie sur la deuxième syllabe.

**Le registre affectif ne doit pas remplacer la précision linguistique.**

## B.8 Mobile-first

La messagerie doit être conçue **prioritairement pour** :

```text
360 × 800
390 × 844
```

Exigences :

* composer toujours visible au-dessus du clavier ;
* safe areas iOS ;
* aucun scroll horizontal ;
* notes vocales accessibles ;
* upload résilient ;
* brouillon conservé lorsque possible ;
* reprise après mise en arrière-plan ;
* état hors ligne ;
* compatibilité PWA et application native future.

## B.9 Permissions

### Professeur ou coach

Peut :

* annoncer ;
* créer un devoir ;
* corriger ;
* répondre ;
* épingler ;
* modérer ;
* voir les remises ;
* clôturer une activité.

### Élève adulte

Peut :

* publier ;
* répondre ;
* remettre une production ;
* réagir ;
* signaler ;

### Profil enfant

* accès au groupe autorisé uniquement ;
* restrictions de pièces jointes ;

### Admin ou centre

**Aucun accès automatique à toutes les conversations.**

Tout accès exceptionnel doit être :

* limité ;
* justifié ;
* journalisé ;

* contrôle des fichiers ;
